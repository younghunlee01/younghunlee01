# 1. IaC 개념
[← 목차로 돌아가기](../README.md)

> Infrastructure as Code — 코드로 인프라를 정의하고 관리하는 방식

## IaC란?

인프라를 콘솔에서 **클릭으로 만드는 대신, 코드로 정의하고 관리**하는 방식입니다.
설정을 파일로 남기기 때문에 동일한 환경을 반복해서 재현할 수 있고, 변경 이력을 Git으로 추적할 수 있습니다.

## 수동 설정 vs IaC

| 구분 | BEFORE — 수동 설정 (Click Ops) | AFTER — IaC (코드로 정의) |
|------|-------------------------------|--------------------------|
| 방식 | 콘솔에서 클릭으로 리소스 생성 | 선언적(Declarative) 방식으로 상태 기술 |
| 재현성 | 같은 환경을 재현하기 어려움 | 동일 환경을 N번이고 재현 가능 |
| 이력 관리 | 변경 이력 추적 불가 | Git으로 변경 이력 관리 |
| 안정성 | 휴먼 에러 발생 가능성 높음 | 리뷰 / CI 파이프라인과 결합 가능 |

## 핵심 키워드

- **Declarative (선언적)** — "어떻게"가 아니라 "무엇을" 원하는지 기술한다. 목표 상태를 적으면 도구가 알아서 그 상태로 맞춘다.
- **Idempotent (멱등성)** — 같은 코드를 몇 번 실행해도 결과가 동일하다. 이미 원하는 상태면 아무 작업도 하지 않는다.
- **Version Control (버전 관리)** — 인프라 변경을 Git으로 관리하고 리뷰할 수 있다.
- **Reproducible (재현성)** — 동일한 인프라를 언제든 똑같이 다시 만들 수 있다.
- **Automation (자동화)** — 반복 작업을 없애고 CI/CD 파이프라인에 연결할 수 있다.

## 왜 중요한가

수동 설정은 사람이 일일이 클릭하기 때문에 실수가 생기기 쉽고, 무엇을 바꿨는지 기록이 남지 않습니다.
IaC는 인프라를 코드로 관리하므로 **재현성 · 이력 추적 · 자동화**를 동시에 확보할 수 있어,
협업 환경과 운영 안정성 측면에서 큰 이점을 가집니다.
